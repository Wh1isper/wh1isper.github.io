# 综述 #

自顶向下的设计过程，就是一个不断拆解问题，提出解决措施，检验措施效果风险的过程。设计过程始终关注需求问题是否已经解决，是否存在实现风险，在一些关键环节，为了防止出现偏差，需要安排沟通、检视、评审等活动，利用团队力量保证设计过程可控，产出不出现重大质量问题。

## 设计过程的三个阶段 ##

1. 需求分析，目标制定
2. 构思方案，分析风险
3. 要点细化，查缺补漏

## 核心原则 ##

- 一条基本原则：以需求为靶心，以风险为准星
- 两个分析过程：需求分析过程，风险分析过程
- 三种纠偏手段：沟通，检视，评审

## 关键活动 ##

1. 需求分析阶段-需求分析
2. 设计方案截断-核心技术点方案选型
3. 设计方法阶段-设计方案
4. 设计方案阶段-风险分析
5. 设计细化阶段-设计细化
6. 方案/设计评审

# 需求分析步骤方法 #

## 需求发散 ##

需求发散一般使用思维导图作为分析工具，以金字塔原则作为分析原则

1. 需求层层分解
2. 上层概括下层
3. 同层独立穷尽

需求发散时从5个维度进行发散

1. 功能
2. 性能
3. 场景
4. API
5. 设计约束
	- 可调式
	- 可测试
	- 可扩展
	- 可复用


有以下注意事项：

- 领域的专有名词，需要对名词进行定义，防止对名词的理解不一致
- 如果有形容词或者两次，需要保证可度量或者可定性
- 发散需求时，细化到不存在不确定性，很容易想清楚如何解决
- 找一个结对伙伴，向他解释需求分析，进行补漏
- 通过追问判断需求真实性——挖掘需求本质，以及更好的解决方案


## 需求收敛 ##

对需求进行聚焦，找到最有风险，需要后续设计过程重点关注的需求项。

需求的聚焦，从重要性、复杂度、影响面三个维度出发， 对各需求项进行标注。然后综合着三个维度的信息，给每个需求项标注“风险”。

**风险越高的需求项，意味着后续需要更多精力投入，更严格的质量控制措施。**

### 重要性 ###

用户或其它模块有多依赖这个模块，标识着出问题时负面影响的深度/程度。

1. 核心需求
2. 重要需求
3. 优化需求
4. 拓展需求

### 复杂度 ###

意味着是否出现问题，复杂度影响出现问题的概率

1. 知识领域有多广、多深？
2. 要求高低？如性能、可靠性
3. ..见技术难度评估

### 影响面 ###

有多少业务依赖本模块提供服务，出问题时负面影响的广度；影响面越大越应该提前考虑。影响面确定后有助于其他人开始并行工作。

    风险值 = 出问题的概率（复杂度）* 出问题的影响程度（重要性） * 出问题的影响广度（影响面）

# 设计方案制定 #

1. 确定方案思路（设计原理）
2. 场景分析
3. 核心需求点的方案选型
4. 使用三视图对需求进行建模
5. 风险分析
6. 方案评审

> 三视图：可被计算机落地实施的具体结构/逻辑/数据

> 方案风险分析：目的是为了确认方案各个环节的质量，检查有没有风险遗漏。

## 要点 ##

拆分！把一个大而复杂的问题，拆解为多个小而简单的问题，分别去独立解决。

- 切蛋糕（垂直拆分——拆开后各组件同构）
- 拆汽车（垂直拆分——拆开后各组件不同构）
- 铲草皮（水平拆分——按知识领域不同拆开）
- 白加黑（按场景拆分）

# 设计细化 #

设计细化：把第二阶段制定的方案细化到可以指导他人编码的程度

1. 规格化：采用三视图呈现关键思路
	1. 模块结构的最终产出是各个子模块的规格定义，以及模块关系的说明
	2. 数据结构的最终产出是给出数据结构定义
	3. API/流程的最终产出是给出函数定义列表

2. 查缺补漏：从可调试/可测试/可实现/可复用/可扩展/安全性/可靠性出发，分析设计方案在实现/调试/测试/服用/扩展/维护等生命周期环节可能会遇到的问题，及提前的准备
3. 检验：从“设计目标达成”，“可读性”，“易实现/可维护性”，“健壮性”，“可调式/可测试”，“可扩展/可复用”等维度考察设计

## 要点 ##

解耦：

1. 识别易变流程和不变流程
2. 构造易变流程和不变流程之间的间接层（作为交互媒介的数据结构）
3. 分离出“制程”阶段，利用“流程描述”定义生产流程，提高生产流程的灵活性 

# 设计三步法 #

1. 整理需求、确定目标
2. 构思方案、风险分析
3. 敲定细节

## 整理需求、确定目标 ##

**将这些内容提炼为一条一条的精简描述，对每一条标识其重要程度，实现难度和关联程度**

对于总设来说，要整理的需求主要来自以下三个方面：

1. 功能需求
2. 性能需求
3. 实现需求（可维护性、可扩展性、可测试性等）

对于概设来说，需要整理的需求主要来自以下三个方面：

1. 总设要求该模块实现的需求内容
2. 总设要求该模块实现的接口
3. 总设要求该模块必须满足的设计约束

## 构思方法、风险分析 ##

一般先通过自顶向下分析方法，从设计需求出发，构思一个或多个整体实现方案。这个方案能囊括所有的需求点。

针对每个方案，可以开展一个风险分析，找出其中风险点，并针对每个风险点，找出风险应对措施。

方案构思阶段可以采用两个视图作为分析工具：结构视图、逻辑视图

通过这两个视图观察系统时，我们需要关注的点：可实现性、完整性、可维护性、可测试性、可扩展性

## 确定细节 ##

重点关注流程、细节，需要输出各类规格说明，确保下一环节的责任人能理解这个设计，没有歧义或模糊不清的内容，保证相关任务能交接到位

最为重要的是可实现性、完整性

对于总设来说，需要给出涉及模块的需求、接口、交互逻辑、设计约束。主要输出是模块定义列表。粒度控制在一个人在一个版本周期内可以完成的范围。

对于概设来说，需要给出函数定义，数据结构定义。主要输出是函数定义列表。规模控制在50行代码以内。
